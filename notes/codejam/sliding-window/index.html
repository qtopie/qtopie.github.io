滑动窗口

February 23, 2025

滑动窗口（Sliding Window） 是一种专门用于处理数组或字符串问题的优化技巧。它的核心思想是将嵌套循环（$O(n^2)$）转化为单层循环（$O(n)$），通过维护一个“窗口”并在数据结构上滑动，避免重复计算。

1. 基本原理

想象你在观察一段长长的铁轨，但你的视线只能透过一扇固定宽度（或可变宽度）的窗户。当你向前移动时，窗户左侧移出的部分不再看，右侧新进入的部分被纳入观察范围。

  • 窗口（Window）：由两个指针（左边界 left 和右边界 right）定义的子序列。
  • 滑动（Sliding）：通过移动指针来更新窗口内容。

2. 算法分类

根据窗口长度是否变化，可以分为两类：

固定窗口 (Fixed Window)

窗口大小 $k$ 是恒定的。通常用于计算“每连续 $k$ 个元素的平均值/总和”。

  • 操作步骤：
  1. 计算前 $k$ 个元素的初始值。
  2. 窗口向右移动一位：加上新进入的元素，减去刚离开的元素。
  3. 重复此过程直到遍历结束。

可变窗口 (Variable Window)

窗口大小根据特定条件动态伸缩。通常用于寻找“满足条件的最短/最长子串”。

  • 操作步骤：
  1. 右指针 right 不断扩大窗口。
  2. 当窗口内的数据满足（或不满足）某个条件时，移动左指针 left 来收缩窗口。
  3. 在这个过程中记录最优解。

3. 典型应用场景

滑动窗口在以下场景中非常高效：

  • 字符串处理：寻找无重复字符的最长子串、字符串排列、最小覆盖子串。
  • 数组计算：子数组最大和、满足条件的最小连续子数组。

4. 算法模板（Python 伪代码）

对于可变窗口，通用的逻辑如下：

func slidingWindow(s string) int {
	left, right := 0, 0
	// window 用于记录窗口内字符出现的次数或其他状态
	window := make(map[byte]int) 
	res := 0

	// 在 Go 中，range 字符串得到的是 rune，但按索引访问得到的是 byte
	for right < len(s) {
		// c 是将移入窗口的字符
		c := s[right]
		// 右移窗口
		right++
		
		// 进行窗口内数据的一系列更新
		// 例如：window[c]++

		// 判断左侧窗口是否要收缩
		// 这里的 windowNeedsShrink 是你的业务逻辑条件
		for windowNeedsShrink {
			// d 是将移出窗口的字符

			// 左移窗口
			left++

			// 例如：window[d]--
		}
		
		// 在这里更新结果 res
	}
	return res
}

5. 为什么它比暴力解法快？

暴力解法通常对每一个可能的子数组都进行遍历计算，时间复杂度往往是 $O(n^2)$。 滑动窗口利用了计算的连续性：既然前一个窗口和当前窗口有大量重合，我们只需要处理边缘的两个元素，而不需要重新计算中间的元素。这使得复杂度直接降到 $O(n)$。

你想了解滑动窗口在具体编程语言（如 Go 或 Python）中的实现代码，还是想看看它在分布式系统流量控制（如 Sentinel 的滑动窗口限流）中的具体应用？

将嵌套循环（$O(n^2)$）优化为单层循环（$O(n)$），其核心数学原理在于减少冗余计算和利用单调性（Monotonicity）。

在暴力解法中，我们通常会对每一个可能的起点 $i$ 遍历每一个可能的终点 $j$，这导致了大量的重复扫描。而 $O(n)$ 的优化（如滑动窗口或双指针）本质上是基于以下几个数学逻辑：

数学解释

1. 消除冗余计算：前缀和与差分思想

在暴力计算子数组和时，我们可能会重复计算中间重叠的部分。

  • 暴力逻辑：计算 $[i, j]$ 的和，再计算 $[i, j+1]$ 的和。
  • 数学优化：

$$Sum(i, j+1) = Sum(i, j) + Array[j+1]$$

通过只处理“增量”（新进入的元素）和“减量”（移出的元素），我们将原本需要 $O(k)$ 时间的求和操作降到了 $O(1)$。

2. 空间的单调性（The Property of Monotonicity）

这是滑动窗口能够工作的最关键原理。假设我们要找满足某个条件 $C$ 的最长子数组：

  • 如果子数组 $[i, j]$ 满足条件 $C$，那么对于任何 $j’ < j$，其子集（在某些特定条件下，如全正数数组）也可能满足条件，但我们不需要重新从 $i+1$ 开始扫描。
  • 关键结论：如果右边界 $right$ 向右移动，左边界 $left$ 绝不需要向左回溯。

由于 $left$ 和 $right$ 都只向右移动，每个元素最多进出窗口各一次，因此：

$$T(n) = \text{进窗口次数} + \text{出窗口次数} = n + n = 2n \implies O(n)$$

3. 搜索空间的修剪（Pruning the Search Space）

从线性代数或集合的角度看，$O(n^2)$ 实际上是在遍历一个二维矩阵的所有坐标 $(i, j)$，其中 $i \le j$。

  • 暴力法：遍历整个三角形区域。
  • $O(n)$ 优化：根据问题的约束条件（例如：当窗口不满足条件时，所有以当前 left 为起点的更长窗口都不可能满足条件），我们直接跳过（修剪）了矩阵中大量的无效坐标。
算法搜索范围数学表现
暴力 (Brute Force)$\sum_{i=1}^{n} (n-i+1)$等差数列求和 $\approx \frac{1}{2}n^2$
优化 (Sliding Window)$left + right \le 2n$线性扫描

4. 状态转移的连续性

在动态规划或滑动窗口中，当前状态 $S_k$ 往往可以由前一个状态 $S_{k-1}$ 通过极小的代价转换而来。这种数学上的递推关系取代了全量的重新计算。

例如，在求“滑动窗口最大值”时，利用双端队列维护单调性，本质上是利用了“如果 $a < b$ 且 $a$ 比 $b$ 先过期，那么 $a$ 永远不可能成为最大值”这一逻辑判断，从而排除了不可能的解空间。

总结来说：$O(n^2)$ 到 $O(n)$ 的跨越，不是因为计算机运行变快了，而是因为我们通过数学推导，证明了大部分计算步骤是“无用功”，并利用指针的单调移动替代了地毯式的搜索。
